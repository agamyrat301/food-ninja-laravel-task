# Сокращатель ссылок (Laravel 10 + Filament v3)

Веб-приложение на Laravel 10, позволяющее зарегистрированным пользователям
создавать короткие ссылки, отслеживать переходы по ним и управлять своими
ссылками через личный кабинет — в двух вариантах: обычный Blade-кабинет и
кабинет на базе админ-панели [Filament v3](https://filamentphp.com/).

## Возможности

- **Аутентификация** — регистрация и вход (Laravel Breeze).
- **Создание коротких ссылок** — пользователь указывает оригинальный URL
  (например `https://example.com/page`) и получает короткую ссылку вида
  `http://ваш-домен/abc123`.
- **Редирект + статистика** — переход по короткой ссылке делает 302-редирект
  на оригинальный URL и записывает переход (IP-адрес, дата и время,
  User-Agent) в таблицу `clicks`.
- **Личный кабинет** (`/links`, Blade) — список своих ссылок с количеством
  переходов, создание новой ссылки, удаление, страница статистики по каждой
  ссылке (список переходов + общее количество кликов).
- **Личный кабинет на Filament** (`/cabinet`) — то же самое, но в виде
  админ-панели: доступен любому зарегистрированному пользователю, каждый
  видит и может редактировать только свои ссылки; пользователь с флагом
  `is_admin` видит и может управлять ссылками всех пользователей.

Обе версии кабинета работают с одними и теми же таблицами и правами доступа —
это просто два разных интерфейса поверх одной модели.

## Стек

- Laravel 10, PHP 8.1+
- MySQL
- Laravel Breeze (Blade) — аутентификация и обычный кабинет
- Filament v3 — второй, admin-panel-based кабинет

## Установка

```bash
composer install
npm install && npm run build

cp .env.example .env
php artisan key:generate
```

Настройте подключение к БД в `.env` (по умолчанию MySQL, база `link_shortener`):

```
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=link_shortener
DB_USERNAME=root
DB_PASSWORD=
```

Создайте базу данных, накатите миграции и засейте тестового администратора:

```bash
php artisan migrate --seed
php artisan serve
```

Приложение будет доступно на `http://127.0.0.1:8000`.

## Вход в Filament-кабинет

Filament-панель находится по пути **`/cabinet`** (не `/admin`), так как это
полноценный личный кабинет для всех пользователей, а не только для
администратора.

- **Страница входа:** `http://127.0.0.1:8000/cabinet/login`
- **Страница регистрации:** `http://127.0.0.1:8000/cabinet/register`
  (можно зарегистрироваться прямо в Filament, либо через обычную форму
  `/register` — это одна и та же таблица пользователей, вход сработает
  в обоих кабинетах).

Сидер (`database/seeders/DatabaseSeeder.php`) создаёт тестового
администратора:

| Email | Пароль | Роль |
|---|---|---|
| `admin@example.com` | `password` | администратор (`is_admin = true`), видит ссылки всех пользователей |

Любой другой зарегистрированный пользователь (`is_admin = false`) при входе
в `/cabinet` увидит и сможет управлять только своими собственными ссылками —
это обеспечивается через `LinkResource::getEloquentQuery()`, а поле выбора
владельца (`user_id`) скрыто от обычных пользователей и принудительно
проставляется на сервере при создании/редактировании, так что подменить
владельца через форму нельзя.

## Обычный (Blade) кабинет

- `http://127.0.0.1:8000/register` — регистрация
- `http://127.0.0.1:8000/login` — вход
- `http://127.0.0.1:8000/links` — список своих ссылок, форма создания
- `http://127.0.0.1:8000/links/{id}` — статистика по конкретной ссылке
  (список переходов, общее количество кликов)

## Как это устроено внутри

Схема данных:

- `links` — `id`, `user_id`, `original_url`, `code` (уникальный короткий код,
  генерируется автоматически при создании), `created_at`/`updated_at`.
- `clicks` — `id`, `link_id`, `ip_address`, `user_agent`, `created_at`
  (момент перехода).

Логика намеренно разнесена по слоям, а не свалена в контроллеры, — оба
кабинета (Blade и Filament) переиспользуют одни и те же классы вместо
дублирования правил:

- **`App\Policies\LinkPolicy`** — единая точка правил доступа
  (`view`/`update`/`delete`: владелец или `is_admin`). Используется и в
  Blade-контроллере (`$this->authorize(...)`), и автоматически в
  Filament-ресурсе (Filament сам вызывает политику модели на каждое
  действие в таблице).
- **`App\Actions\Links\CreateLinkForUser`** — единственное место, где
  реально создаётся `Link`. И `LinkController::store`, и
  `LinkResource\Pages\CreateLink` вызывают этот же action, поэтому
  поведение создания ссылки гарантированно одинаковое в обоих кабинетах.
- **`App\Contracts\ShortCodeGenerator`** + **`App\Services\RandomShortCodeGenerator`**
  — генерация кода вынесена за интерфейс и резолвится через контейнер
  (`AppServiceProvider::register()`), а не жёстко зашита в модель. Это
  позволяет позже подменить стратегию (например, на последовательный
  base62) не трогая `Link` и контроллеры.
- **`App\Http\Requests\StoreLinkRequest`** — валидация вынесена из
  контроллера в отдельный Form Request.
- **`App\Events\LinkVisited`** + **`App\Listeners\RecordLinkClick`** —
  `RedirectController` не создаёт запись в `clicks` напрямую, а только
  диспатчит событие «ссылку посетили»; сам факт записи клика — это
  отдельный слушатель. Это разделение позволяет добавить, например,
  очередь для гео-аналитики по клику, не трогая контроллер редиректа.
- `GET /{code}` (`App\Http\Controllers\RedirectController`) — публичный
  маршрут без авторизации, объявлен последним в `routes/web.php`, чтобы не
  перехватывать `/login`, `/register`, `/links`, `/cabinet` и т. д.
- `App\Http\Controllers\LinkController` — тонкий контроллер Blade-кабинета
  (`index`, `store`, `show`, `destroy`): валидация в Form Request,
  авторизация в Policy, создание — в Action.
- `App\Filament\Resources\LinkResource` +
  `RelationManagers\ClicksRelationManager` — тот же домен в виде
  Filament-ресурса; `getEloquentQuery()` скрывает чужие ссылки от обычных
  пользователей, а `Pages\CreateLink`/`Pages\EditLink` не доверяют полю
  `user_id`, пришедшему из формы, для не-админов.

## Тесты

```bash
php artisan test
```

41 тест (PHPUnit, sqlite in-memory — `phpunit.xml`), покрывают:

- регистрацию/вход (стандартные тесты Breeze);
- `LinkManagementTest` — создание, валидацию URL, список только своих
  ссылок, просмотр статистики и удаление, включая проверку 403 при попытке
  доступа к чужой ссылке;
- `RedirectTest` — редирект по короткому коду, диспатч `LinkVisited`,
  запись клика с IP и User-Agent, 404 на несуществующий код;
- `Filament/LinkResourceTest` — обычный пользователь видит и может
  создавать только свои ссылки (в том числе при попытке подменить
  `user_id` через форму), администратор видит и назначает ссылки всем;
- `RandomShortCodeGeneratorTest` — юнит-тест на саму стратегию генерации
  кода.
